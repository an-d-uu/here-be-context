# Maps 🗺️

Maps explain how the pieces fit together.

This directory contains relationship documents that connect frameworks, lexicons, guides, and other repo artifacts without collapsing them into one oversized doctrine scroll.

If a guide tells you how to use one tool, a map tells you how multiple tools relate, where their borders are, and which one gets the steering wheel when they seem to overlap.

---

## 🧠 What Lives Here

This directory is for interpretive and connective documents such as:

* system maps
* framework relationship guides
* precedence maps
* source-role references
* usage-order documents
* collision notes between adjacent artifacts
* cross-directory connection files

A map does not usually introduce a new behavior system.
It explains how existing ones work together.

---

## 🎯 What This Directory Is For

Use `maps/` when the real problem is not a missing instruction, but a missing relationship.

This is especially useful when:

* two or more sources address nearby concepts from different angles
* a repo has grown enough that files can be individually clear but collectively muddy
* a framework needs a clear place in the larger system
* a lexicon, guide, or quest works well on its own but creates ambiguity when combined with others
* you need to define sequence, scope boundaries, or precedence without rewriting the source files themselves

A good map reduces collision.
A great map makes the repo easier to trust.

---

## ⚠️ What This Directory Is Not

This is not:

* a duplicate copy of the source documents
* a lexicon folder in different armor
* a place for general notes with no connective job
* a substitute for writing a clear guide in the first place
* a dumping ground for anything that feels vaguely strategic

If a file does not explain how multiple pieces relate, it probably belongs somewhere else.

---

## 🗺️ Relationship to the Rest of the Repo

* **skill-trees/** teach reusable mechanics
* **quests/** define project-level behavior
* **relics/** shape system-wide behavior
* **encounters/** document unstable or disruptive inputs
* **guides/** explain how to use a specific framework, method, or tool
* **lexicons/** control wording, phrasing, and expression
* **maps/** explain how those pieces connect, sequence, and constrain one another

If guides teach the local terrain,
`maps/` shows where the roads cross.

---

## 🛠️ How to Use These Files

Each map should help answer questions like:

* What is each source for?
* Where does one source stop and another begin?
* In what order should these be applied?
* Which source should win when two seem related?
* What confusion patterns should be expected?
* What layer of the system does each artifact belong to?

Maps are not decorative overviews.
They are coordination tools.

Use them when the repo has enough moving parts that correct files can still produce confused outcomes.

---

## 📎 Naming Guidance

Files in this directory should use names that describe the system or relationship being mapped.

Examples:

* `prioritization-system-map.md`
* `language-control-map.md`
* `instruction-precedence-map.md`
* `project-context-map.md`
* `response-layering-map.md`

The filename should tell you what network of ideas the file is connecting.

---

## 🧭 Good Map Behavior

A strong map should:

* stay shorter than the combined documents it connects
* clarify scope without rewriting everything
* name conflicts directly
* define order when order matters
* reduce interpretation drift across related files

If a map becomes longer than the territory it is mapping, the cartographer has gone feral.

---

A repo can hold many sharp tools.
A map keeps them from all claiming to be the hammer.
