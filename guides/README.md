# Guides 🧭

Guides explain how to use a specific tool, framework, or method.

This directory contains practical reference documents for systems you want to apply deliberately, not vaguely admire from across the tavern.

A guide is where a concept stops being interesting and starts becoming usable.

---

## 🧠 What Lives Here

This directory is for applied working documents such as:

* framework guides
* method playbooks
* operating guides
* decision rubrics
* usage instructions for a specific model or process
* practical references that translate abstract ideas into repeatable use

A guide should help someone do the thing, not just recognize its name in the wild.

---

## 🎯 What This Directory Is For

Use `guides/` when you want a document that turns a concept into a working tool.

This is especially useful when:

* an external framework needs translation into your actual environment
* you want a repeatable way to handle common situations
* a model or method needs clear definitions, response paths, and anti-patterns
* a source should help make decisions, not merely explain theory
* a reusable operating reference needs to sit somewhere more concrete than a quest and less broad than a relic

A good guide reduces improvisation.
A great guide reduces bad improvisation.

---

## ⚠️ What This Directory Is Not

This is not:

* a project instruction folder
* a system-prompt archive
* a lexicon directory with extra paragraphs
* a map of relationships between multiple sources
* a scrapbook of productivity ideas with no operational value

If a file does not help someone apply a specific framework or method, it probably does not belong here.

---

## 🗺️ Relationship to the Rest of the Repo

* **skill-trees/** teach reusable mechanics
* **quests/** define project-level behavior
* **relics/** shape system-wide behavior
* **encounters/** document unstable or disruptive inputs
* **lexicons/** control wording and expression
* **maps/** explain how multiple pieces work together
* **guides/** explain how to apply one framework, method, or tool in practice

If a map tells you how systems relate,
`guides/` tells you how to actually use one without wandering into a bog.

---

## 🛠️ How to Use These Files

Each guide should help answer questions like:

* What is this framework for?
* When should it be used?
* When should it not be used?
* How does it translate into real decisions or actions?
* What common mistakes should be avoided?
* How does it fit with the other tools already in use?

Guides should be practical enough to reference mid-task, not so theoretical that they require a second guide to explain the first.

---

## 📎 Naming Guidance

Files in this directory should use names that describe the framework, method, or tool being applied.

Examples:

* `eisenhower-matrix-work-guide.md`
* `meeting-triage-guide.md`
* `scope-control-guide.md`
* `rewrite-decision-guide.md`
* `intake-rubric-guide.md`

The filename should tell you what the guide helps you use.

---

## 🧭 Good Guide Behavior

A strong guide should:

* define the purpose clearly
* explain when to use it and when not to
* translate abstract ideas into working rules
* include anti-patterns or common failure modes
* fit cleanly with related maps, lexicons, or project instructions

If a guide cannot survive contact with real work, it is just motivational wallpaper with better formatting.
